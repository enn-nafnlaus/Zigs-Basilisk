Issue: Multi-Module Panic and Crash Vectors


Primary issue: std.zip.extract panics when verify_checksums=true

Severity: Medium (crashes user programs that use a documented API option)

The std.zip.extract function accepts an ExtractOptions struct with a verify_checksums: bool field (defaulting to false). If a user sets this to true, the function immediately panics with @panic("TODO unimplemented") instead of either implementing the feature or returning an error.

Problem:

  1. The option is part of the public API (it's a field in a public struct).
  2. The default is false, so users who want checksum verification will set it to true.
  3. The panic happens unconditionally — there's no way to get checksum verification to work.
  4. The panic occurs before the archive is parsed, so even an empty or invalid input triggers it and the caller's catch cannot recover.
  5. Calling Iterator.Entry.extract directly bypasses this check and silently ignores verify_checksums, so both public extraction paths violate the option's apparent contract in different ways.

File: lib/std/zip.zig, line 655

Code:

pub fn extract(dest: Io.Dir, fr: *File.Reader, options: ExtractOptions) !void {
    if (options.verify_checksums) @panic("TODO unimplemented");

Test case (de minimis):

// Requires a file "test.zip" in the working directory.
// Creates it with: echo hello > test.txt && zip test.zip test.txt
const std = @import("std");
pub fn main(init: std.process.Init) !void {
    const dir = std.Io.Dir.cwd();
    var file = try dir.openFile(init.io, "test.zip", .{});
    defer file.close(init.io);
    var buf: [4096]u8 = undefined;
    var reader = file.reader(init.io, &buf);
    var dest = try dir.openDir(init.io, "out", .{});
    defer dest.close(init.io);
    // Panics with "TODO unimplemented":
    std.zip.extract(dest, &reader, .{ .verify_checksums = true }) catch |err| {
        std.debug.print("Got error: {}\n", .{err});
    };
}

Test result:

program panics with:

thread panic: TODO unimplemented
/usr/lib/zig/std/zip.zig:655:35: in extract (std.zig)
    if (options.verify_checksums) @panic("TODO unimplemented");

Checksum verification is not merely unavailable when the option is enabled; extraction with its documented default (false) accepts payloads which do not match the archive's recorded CRC. A stored ZIP containing UNIQUE_PAYLOAD_12345 was modified in-place to contain UNIQUE_PAYLOAD_X2345, leaving the recorded CRC unchanged. unzip -t rejected it with:

bad CRC db3c2719  (should be f4c7798b)

The same archive was accepted by std.zip.extract with default options (zig_exit=0), and the corrupted bytes were written to disk.

The practical hazard depends on how the extracted files are used. Corrupted executables, scripts, configuration, firmware, or build inputs could later be executed or consumed, so silently accepting corruption can have serious consequences in a pipeline which assumes CRC verification occurred.

How to fix:

Either implement checksum verification (compute CRC32 of the decompressed data and compare against the CRC stored in the local file header / central directory), or remove the verify_checksums field from ExtractOptions and document that checksum verification is not yet supported. At minimum, return an error.Unsupported instead of panicking.

Issue #2: std.coff.SectionHeader.getNameOffset panics on malformed section names

Severity: Low-Medium (externally controlled denial of service during section-name resolution)

The getNameOffset function in std.coff.SectionHeader parses the string offset from a COFF section name that starts with /. It uses std.fmt.parseInt(u32, ..., 10) catch unreachable, which panics if the characters after / are not valid decimal digits.

A valid COFF long section name looks like /123 (offset into the string table). A malformed name like /abc causes parseInt to return error.InvalidCharacter, which is caught by catch unreachable and converted to a panic.

Problem:

  1. COFF files are binary data from external sources (object files, archives).
  2. A corrupted or maliciously crafted COFF file can cause the developer's program to panic in a place where a panic is not anticipated as a possibility.
  3. The function is in the public std.coff API — any Zig program parsing COFF files is affected.
  4. catch unreachable is inappropriate for parsing untrusted input.

File: lib/std/coff.zig, line 476

Code:

pub fn getNameOffset(self: SectionHeader) ?u32 {
    if (self.name[0] != '/') return null;
    const len = std.mem.findScalar(u8, &self.name, @as(u8, 0)) orelse self.name.len;
    const offset = std.fmt.parseInt(u32, self.name[1..len], 10) catch unreachable;
    return offset;
}

Test case (de minimis):

Test result:

program panics with:

thread panic: attempt to unwrap error: InvalidCharacter
/usr/lib/zig/std/coff.zig:476:75: in getNameOffset (std.zig)
    const offset = std.fmt.parseInt(u32, self.name[1..len], 10) catch unreachable;

Normal parser API test: a bounded 68-byte AMD64 COFF object was constructed with one section named /abc, zero symbols, and a valid four-byte empty string table. This path also aborts with exit status 134:

Coff.init
  -> Coff.getSectionByName(".text")
  -> Coff.getSectionName
  -> SectionHeader.getNameOffset
  -> panic: attempt to unwrap error: InvalidCharacter

This proves that callers do not need to construct SectionHeader manually. An internal consumer exists in std.debug.SelfInfo.Windows, which resolves DWARF section names through getSectionByName; normally it parses the running program's own image rather than arbitrary network input.

How to fix:

changing this one line to catch return null is insufficient, because Coff.getSectionName immediately unwraps the optional with .?. Return a typed malformed-name error and propagate it through getSectionName and getSectionByName. Also validate the offset and NUL termination: Strtab.get currently asserts that the offset is in range and then performs a sentinel scan not bounded to the string-table slice. Test /abc, /, decimal overflow, out-of-range offsets, and unterminated strings together.

Issue #3: src/link/C.zig panics on AnalysisFail for anonymous declarations

Severity: Low (reliable compiler crash, but limited to the non-default C backend)

When the C backend fails to codegen an anonymous declaration (UAV), the linker code in src/link/C.zig panics with @panic("TODO: C backend AnalysisFail on anonymous decl") instead of propagating the error.

The commented-out code shows the intended error handling path:

//try zcu.failed_decls.put(gpa, decl_index, object.dg.error_msg.?);
//return;

This is a real bug because valid Zig source can reliably reach the path. The same unsupported value produces a normal C-backend diagnostic when stored in a named declaration, but aborts the compiler when stored in an anonymous declaration.

File: src/link/C.zig, line 260

Code:

codegen.genDeclValue(&object, c_value.constant, c_value, alignment, .none) catch |err| switch (err) {
    error.AnalysisFail => {
        @panic("TODO: C backend AnalysisFail on anonymous decl");
        //try zcu.failed_decls.put(gpa, decl_index, object.dg.error_msg.?);
        //return;
    },
    error.WriteFailed, error.OutOfMemory => return error.OutOfMemory,
};

Test case:

const U = extern union {
    integer: u32,
    float: f32,
};

export fn getUnion() *const U {
    return &@as(U, @bitCast(@as(u32, 0x3f800000)));
}

Command:

zig build-obj test.zig -ofmt=c -fno-emit-bin -femit-bin=test.c

Test result with installed Zig 0.16.0:

CONFIRMED. The compiler aborts with exit status 134:

thread panic: TODO: CBE error.AnalysisFail on uav

Control test: assigning the same value to const value: U and returning &value exits normally with a compile error:

error: TODO: C backend: implement extern union backing type rendering in static initializers

The source checkout has an additional problem that can make this worse. DeclGen.fail obtains a source location using dg.pass.nav, while updateUav sets dg.pass = .{ .uav = uav }. Since every C-codegen AnalysisFail is created by DeclGen.fail, a safety-enabled build of this checkout may panic on accessing the inactive .nav union field before it even reaches the explicit panic in updateUav.

How to fix:

this is not as simple as uncommenting the stale failed_decls code: UAVs do not have a declaration index or source location. Make DeclGen.fail choose a valid source location for .uav, then record/report the failure through an error collection keyed appropriately for a UAV. The flush must skip the failed UAV and return normal compilation failure rather than aborting.

Issue #4: LZMA decompressor discard() panics with "TODO"

Severity: Low (crashes programs that try to skip data in LZMA streams)

The LZMA decompressor's discard function (used to skip decompressed data without reading it) panics with @panic("TODO") instead of either implementing the skip or returning an error.

File: lib/std/compress/lzma.zig, line 700

Code:

fn discard(r: *Reader, limit: std.Io.Limit) Reader.Error!usize {
    const d: *Decompress = @alignCast(@fieldParentPtr("reader", r));
    _ = d;
    _ = limit;
    @panic("TODO");
}

Test case:

CONFIRMED with the valid "Hello world\n" fixture already present in lib/std/compress/lzma/test.zig:

var input: std.Io.Reader = .fixed(&compressed);
var d = try std.compress.lzma.Decompress.initOptions(
    &input, init.gpa, &.{}, .{}, std.math.maxInt(u32),
);
defer d.deinit();
try d.reader.discardAll(1);

The process aborts with exit status 134 and this call chain:

lzma.discard -> Reader.discardShort -> Reader.discardAll

The affected public operations include discard, discardShort, discardAll, discardAll64, and discardRemaining whenever they must consume beyond already-buffered bytes. Ordinary read/stream operations do not implicitly call this callback, which keeps the severity Low.

The implementation violates the generic Reader contract unnecessarily: Reader.VTable.discard already defaults to Reader.defaultDiscard. The adjacent XZ decompressor contains the same TODO panic and should be fixed and tested at the same time.

How to fix:

remove the custom callback and use Reader.defaultDiscard, or delegate to it explicitly. Returning error.Unimplemented is not possible via Reader.Error and would not satisfy the Reader contract.

Concern: print_value.zig has panic handlers for error.

Severity: Risk, but not confirmed exploitable

The print_value.zig module handles formatting Value objects for diagnostics and debug output. Both format adapters panic if print returns error.Canceled.

The comment in the code explicitly acknowledges this:

error.Canceled => @panic("TODO"), // pls stop returning this error mlugg

Source tracing shows that print can only acquire error.Canceled through operations which may resolve types while formatting:

  1. lazy integer formatting calls Type.abiAlignmentSema or abiSizeSema;
  2. pointer formatting calls Value.pointerDerivationAdvanced and may resolve field alignments/types when a Sema is supplied.

Those operations use the broad Zcu.SemaError/CompileError set, which includes Canceled. There are real cancellation-producing I/O operations in Sema (for example @embedFile ultimately propagates Io.Cancelable), so the branch is not type-system-impossible. However, no concrete route was found that cancels exactly while one of these formatting-time resolution operations is active. Normal command-line compilation does not request cancellation, and the incremental compiler server processes update requests serially rather than canceling an old update when a new one arrives. In short, this does not appear to be a current risk, but remains a potential snag for future development.

File: src/print_value.zig, lines 30 and 40

Code:

pub fn formatSema(ctx: FormatContext, writer: *Writer) Writer.Error!void {

        error.OutOfMemory => @panic("OOM"),
        error.ComptimeBreak, error.ComptimeReturn => unreachable,
        error.AnalysisFail => unreachable,
        error.Canceled => @panic("TODO"), // pls stop returning this error mlugg

    };
}

There is, however, one final structural issue regarding these reports.
